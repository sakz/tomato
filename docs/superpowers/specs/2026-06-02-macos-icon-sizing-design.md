# Tomato macOS 图标视觉占比调整设计

## 概述

当前 `Tomato` 的 macOS 应用图标在 Dock 和启动台中的主体视觉占比偏大。问题不在于系统缩放异常，而在于图标前景主体在圆角矩形背景中的占用面积过高，导致它和常见 macOS 应用图标相比显得更“满”、更大。

本次调整目标是参考苹果官方 `macOS` App Icon 的一致性原则，在保持现有品牌风格、配色和背景形态不变的前提下，降低番茄主体的视觉占比，让图标与系统中其他正常应用的感知大小更接近。

## 背景与现状

- 项目是 `Tauri v2` 桌面应用，打包配置位于 `src-tauri/tauri.conf.json`
- macOS 应用图标当前来自 `src-tauri/icons/icon.icns`
- 现有主源图标位于项目根目录 `app-icon.svg`
- 当前源图中背景圆角矩形铺满画布，属于合理的 macOS 图标外轮廓
- 偏大的主要原因是番茄本体、高光和叶片整体过于接近背景边缘，留白不足

## 设计目标

- 让 macOS 图标在 Dock 和启动台中的感知大小回到常见系统应用水位
- 保留现有深色背景、红色番茄和顶部深色叶片的品牌识别
- 不修改 `Windows`、`Android`、`iOS` 现有图标资源
- 不调整 `tauri.conf.json` 中的图标引用路径，避免影响现有打包配置

## 非目标

- 不重做一套新的品牌图标
- 不同步重生成其他平台图标
- 不改网页 favicon 或前端页面中的图标资源
- 不引入 Xcode Icon Composer 或新的跨平台图标设计体系

## 方案对比

### 方案 A：仅缩小 macOS 图标前景主体

做法：保留背景圆角矩形不变，仅缩小番茄本体、高光和叶片的尺寸，并微调其在画布中的垂直位置，使四周留白更接近 macOS 常见图标视觉比例。

优点：

- 直接解决“看起来比别的软件大”的问题
- 改动最小，不破坏现有品牌风格
- 不影响其他平台资源和配置

缺点：

- 需要重新导出一份 macOS 专用素材

### 方案 B：整体缩小整张图标

做法：连背景一起缩小后再导出 macOS 图标。

优点：

- 实现简单

缺点：

- 背景外轮廓也会变小，反而不符合 macOS 图标常见的满版容器外观
- 容易让图标显得“框太小”，而不是主体大小正常

### 方案 C：重做专用 macOS 图标构图

做法：基于苹果模板重画一版更偏系统风格的图标。

优点：

- 可获得最精细的系统一致性

缺点：

- 超出当前需求范围
- 成本高，且会增加多平台视觉分叉

## 选定方案

采用方案 A：新增或派生一份 macOS 专用源图，仅对前景主体进行缩放与位置微调，然后重新导出 `src-tauri/icons/icon.icns`。

## 视觉设计细则

### 保持不变的部分

- 1024 x 1024 画布尺寸
- 深色圆角矩形背景
- 当前的红色番茄渐变风格
- 顶部深色叶片造型
- 整体暗色、简洁、轻拟物的视觉语言

### 调整的部分

- 缩小番茄本体的半径，使主体离背景边缘更远
- 缩小高光椭圆，使其继续贴合缩小后的番茄表面
- 缩小并略微上移叶片，避免叶片与番茄组合后视觉重心过高
- 必要时轻微上移或下移番茄主体，以维持整体居中感

### 视觉判断标准

- 图标单独看时仍然有明确识别度
- 放在 Dock 或启动台环境中时，不再明显比周围常见应用更大
- 主体不接近背景四角，保留更自然的呼吸感
- 缩小后不出现“太空”或“主体过小”的反向问题

## 实现设计

### 文件策略

- 保留 `app-icon.svg` 作为现有通用主源，不直接覆盖，避免影响其他平台后续重导出
- 新增一个 `macOS` 专用源文件 `app-icon-macos.svg`
- 使用该 macOS 专用源图重新生成 `src-tauri/icons/icon.icns`
- 不修改 `src-tauri/tauri.conf.json`，继续引用现有 `icons/icon.icns`

### 导出策略

- 导出范围仅限 `src-tauri/icons/icon.icns`
- 不批量覆盖 `src-tauri/icons` 下其他平台图标资源
- 若工具链支持从 SVG 单独生成 `.icns`，优先走单文件导出
- 若工具链只能先生成一组 PNG 再合成 `.icns`，也仅产出 macOS 所需文件

## 风险与应对

### 风险 1：缩小不足

现象：调整后仍然比其他应用显大。

应对：采用保守但明确可见的缩放幅度，不做肉眼难辨的小调整。

### 风险 2：缩小过度

现象：图标主体显得太小，丢失识别度。

应对：以 Dock 环境下的感知尺寸为准，优先保证与系统应用相近，而不是追求最大留白。

### 风险 3：误伤其他平台图标

现象：通用源文件被直接覆盖后，未来重导出会影响其他平台。

应对：拆分出 macOS 专用源文件，避免跨平台资源被连带修改。

## 验证方案

- 检查 `git diff`，确认修改仅集中在设计文档、macOS 专用图标源和 `src-tauri/icons/icon.icns`
- 校验 `src-tauri/tauri.conf.json` 未发生图标路径变更
- 重新构建或执行图标生成步骤，确保 `.icns` 文件有效
- 若可在本机安装构建产物，则以 Dock / 启动台中的主观视觉比对作为最终验收标准

## 测试策略

- 本次不新增自动化测试
- 以资源文件差异检查和构建验证为主
- 如构建链路包含图标生成命令，则执行一次命令确认无报错

## 交付结果

完成后将得到以下结果：

- 一份 macOS 专用图标源文件
- 更新后的 `src-tauri/icons/icon.icns`
- 保持不变的其他平台图标资源与现有打包配置
